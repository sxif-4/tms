# Hotel Module — Implementation Summary

This document records what was built for the **Hotel Module** frontend, based on [hotelPlan.md](./hotelPlan.md), [frontend-guideline.md](../frontend-guideline.md), and [requirements.md](../requirements.md).

**Status:** Frontend complete (mock data). Backend hotel API not implemented yet.

**Implemented by:** Wildhan  
**Date:** June 2026

---

## 1. Scope

| In scope | Out of scope |
|---|---|
| Hotel staff dashboard UI | NestJS hotel API / Drizzle schemas |
| Rooms CRUD (mock) | `/dashboard/hotel/settings` page |
| Bookings create + status management (mock) | Real persistence to SQLite |
| Promotions CRUD (mock) | `@repo/ui`, `@repo/utils` packages (not wired in this repo) |
| Occupancy / reports view (mock) | Visitor-facing hotel booking flow |

The module uses **in-memory mock data** in `features/hotel/server.ts`. All CRUD operations work in the UI and show toast feedback. When the backend is ready, only `server.ts` needs to be swapped to real `apiFetch` calls (same pattern as `features/users/server.ts`).

---

## 2. Routes

Routes live under `/dashboard/hotel/` (nested in the shared dashboard layout with sidebar + auth guard).

| Route | Page | Description |
|---|---|---|
| `/dashboard/hotel/` | Dashboard | Stat cards + recent bookings table |
| `/dashboard/hotel/rooms` | Rooms | Room list, filter by status, add/edit/delete |
| `/dashboard/hotel/bookings` | Bookings | Booking list, create reservation, confirm/cancel |
| `/dashboard/hotel/promotions` | Promotions | Promotion list, create/edit/delete |
| `/dashboard/hotel/reports` | Reports | Occupancy rate, status breakdown, bookings by room type |

Route files are thin wrappers (`createFileRoute` + loader + page component). `routeTree.gen.ts` is auto-generated by TanStack Router on dev/build.

**Note:** The original plan listed paths like `/hotel/dashboard`. The implementation follows the existing app convention: `/dashboard/hotel/*`.

## 4. Features by Page

### Dashboard (`/dashboard/hotel/`)
- Four stat cards: total rooms, occupied count, active bookings, confirmed revenue
- Recent bookings table (last 5) with guest, room, dates, amount, status badge

### Rooms (`/dashboard/hotel/rooms`)
- Filterable table by status (`available`, `occupied`, `maintenance`)
- Add room via sheet (room number, type, status)
- Edit room inline
- Delete room (blocked if active bookings exist on that room)

### Bookings (`/dashboard/hotel/bookings`)
- Filterable table by status (`pending`, `confirmed`, `cancelled`)
- Create booking sheet: guest name/email, room select, check-in/out date pickers, guest count
- Confirm pending bookings; cancel any non-cancelled booking
- Room status updates automatically on create/confirm/cancel

### Promotions (`/dashboard/hotel/promotions`)
- Table of all promotions with code, discount, validity, active/inactive badge
- Create/edit dialog: name, description, coupon code, percentage or fixed discount, min spend, date range, active toggle
- Delete promotion

### Reports (`/dashboard/hotel/reports`)
- Occupancy rate, available rooms, total revenue summary cards
- Booking status breakdown (confirmed / pending / cancelled) with progress bars
- Bookings by room type (Standard, Deluxe, Suite)

## 6. UI & Patterns

Follows [frontend-guideline.md](../frontend-guideline.md):

- **Components:** shadcn/ui (`Card`, `Button`, `Input`, `Badge`, `Sheet`, `Dialog`, `Select`)
- **Data fetching:** TanStack Query (`useSuspenseQuery`, `useMutation`) via `createServerFn`
- **Forms:** Local `useState` + Zod validation inside mutations
- **Feedback:** Sonner toasts on success/error
- **Theme:** Red primary actions (`bg-red-700`), dark/light mode via existing CSS variables
- **Status badges:** Green = available/confirmed, amber = pending/occupied, red = cancelled/maintenance

## 7. Auth & Testing

**Role:** `hotel_staff`  
**Test account:** `hotel@example.com` / `ChangeMe123!`

After login, the user is redirected to `/dashboard/hotel/` (see `features/auth/redirects.ts`).

**Run locally:**

```sh
# From repo root (tms/)
npm install
npm run dev

# Backend first-time setup (if not done)
cd apps/backend
npm run db:migrate
npm run db:seed
```

- Frontend: http://localhost:3000  
- Backend: http://localhost:4000/api/v1  

If `turbo dev` fails silently, run apps directly:

```sh
cd apps/frontend && npm run dev
cd apps/backend && npm run dev
```

Backend requires `apps/backend/.env` with at least `JWT_SECRET` (32+ chars). See `context/authPlan.md` for full env vars.
